# Hybrid Image Code

**Status:** active research/design note feeding RFC-0006 and ADR-0018  
**Stage:** post-v0.3 prerelease closeout, pre-image-route reframe  
**Purpose:** capture the image-route correction from "scene-spec JSON as the terminal image IR" toward a hybrid architecture where `Semantic IR` remains visible and useful, while `Visual Seed Token` becomes the primary image research layer.

This note is not doctrine by itself. It is a consolidation surface for the current correction pass so maintainers and agents can reason from one document instead of reconstructing the path from chat logs, PRs, and scattered brief amendments.

Canonical decisions still require the engineering lane:

```text
brief / research note -> RFC -> ADR -> exec-plan -> code
```

---

## TL;DR

Wittgenstein should not frame image generation as a binary choice between:

- `structured JSON scene spec`, or
- `direct full VQ token grid`.

The better target is a **hybrid image code container**:

```text
User Prompt
-> Semantic IR
-> Visual Seed Token / Visual Seed Code
-> optional coarse/full VQ hints
-> Seed Expander / Adapter
-> full decoder-native token grid
-> frozen decoder
-> PNG
```

This keeps the current repo's strengths:

- schema-first contracts,
- inspectable manifests,
- frozen decoder doctrine,
- no diffusion / no second image path,
- separable harness / codec / decoder layers,

while correcting the drift where the image route risks collapsing into:

```text
prompt -> longer JSON prompt -> another model does the real image work
```

That collapse is exactly what the project should avoid.

---

## What changed in our understanding

The canonical image wording in the repo currently says:

```text
LLM -> structured JSON scene spec -> adapter -> frozen decoder -> PNG
```

That path is still useful, but its interpretation needs to change.

### What stays true

- `Semantic IR` is valuable.
- The LLM should still organize semantics, composition, palette, and constraints.
- The image route should still target a frozen decoder, not a diffusion generator.
- The adapter should remain small.

### What changes

- `Semantic IR` is **not** the terminal research object for image.
- `Visual Seed Token` / `Visual Seed Code` becomes the **primary image research layer**.
- The adapter is redefined primarily as a **seed expander / visual-code compiler**, not a scene-description reader.
- Two legal output lanes exist:
  - `one-shot hybrid` (default, lighter)
  - `two-pass compile` (high-quality lane)

## Architectural correction

### Old interpretation

```text
User Prompt
-> SceneSpec JSON
-> adapter guesses latent codes
-> decoder reconstructs PNG
```

This is useful as a conservative scaffold, but it over-centers semantic JSON and under-centers the actual image-code thesis.

### Corrected interpretation

```text
User Prompt
-> model-side semantic digestion
-> explicit visual seed code
-> optional coarse/full token hints
-> seed expansion
-> decoder-native token grid
-> frozen decoder
-> PNG
```

Here:

- `Semantic IR` is the model-visible and user-visible layer.
- `Visual Seed Code` is the decoder-facing research object.
- `coarse/full VQ hints` are increasingly direct routes into decoder space.

---

## Why Semantic IR still matters

The correction is **not** "remove JSON" or "remove IR."

`Semantic IR` still earns its place because it does three jobs well:

1. **Model-side semantic organization**
   - lets the LLM unpack user language into structured concepts
   - exposes subject / parts / composition / palette / constraints explicitly

2. **User-facing inspection**
   - gives humans something interpretable to review in the CLI or UI
   - makes failure analysis much easier than opaque token-only output

3. **Auxiliary conditioning**
   - can still be consumed by the seed expander as structured side information
   - should be passed as blocks, not stringified back into prompt text

The key rule is:

> `Semantic IR` remains useful, but it must not be treated as the final image-language contract.

---

## Why Visual Seed Token must become first-class

The project thesis is more interesting when the LLM participates directly in visual coding rather than stopping at a semantic description.

`Visual Seed Token` should therefore be promoted from "future idea" or "escape hatch" to **first-class image output**.

This means:

- it appears in the schema explicitly,
- it appears in prompts / preambles explicitly,
- it appears in evaluation explicitly,
- it appears in manifests explicitly,
- it is treated as the main image research object.

### Desired role

`Visual Seed Token` is:

- short,
- linear or compact,
- generated by LLM next-token prediction,
- closer to decoder token space than natural-language IR,
- still simpler and cheaper than direct full-grid prediction.

### Not desired

`Visual Seed Token` should **not** be:

- merely another prose field hidden inside JSON,
- a giant hand-authored symbolic art vocabulary,
- a disguised prompt,
- a second image model.

## Relevant research priors

The shift toward a first-class seed layer is not a free-form invention. It is supported by several already-relevant lines of prior art:

### 1. TiTok-style compact tokens

TiTok shows that image representations can be made dramatically shorter than the classical 2D VQ grid while still remaining useful for reconstruction and generation. The practical lesson for Wittgenstein is:

- seed representations do not need to be full 32x32 token grids,
- compact token sequences can still carry high-level image structure,
- seed length becomes a tunable design dimension rather than a fixed doctrinal constant.

### 2. FlexTok-style elastic prefixes

FlexTok strengthens the "seed" idea by making the token sequence length itself elastic. Any prefix can represent a lower-fidelity but still meaningful image state.

This matters because it suggests a very natural runtime ladder:

```text
shorter seed -> cheaper / rougher
longer seed -> more expensive / more faithful
```

That is much closer to the Wittgenstein thesis than committing early to one static token budget.

### 3. VAR-style next-scale prediction

VAR shows that raster-order token generation is not the only plausible autoregressive geometry. Coarse-to-fine, next-scale prediction is often a better abstraction than "token 1, token 2, token 3..."

For Wittgenstein this means:

- `Visual Seed Token` may be best understood as a coarse-scale map,
- seed expansion may be a scale-expansion problem rather than a flat token-fill problem,
- the adapter should not be conceptually locked to raster-order token prediction.

### 4. RQ-VAE-style residual code stacks

RQ-VAE provides another seed-expansion geometry: early residual levels hold coarse structure, later residual levels refine detail.

This is a strong match for the "small code -> larger code" intuition because it gives a principled way to say:

```text
seed = early residual structure
expansion = fill later residual layers
```

### 5. SPAE-style lexical visual tokens

SPAE is valuable because it suggests a route where the LLM does not need to natively know a decoder-private numeric codebook. Instead, it can emit a more language-friendly visual code layer that later maps into decoder space.

This is especially relevant to Wittgenstein because one of the main architectural questions is:

> should the LLM emit decoder-native indices directly, or emit a more LLM-natural visual seed language first?

SPAE strengthens the case for the second option.

### Research implication

These priors collectively argue for the same broad move:

- keep semantic IR,
- add a true seed layer,
- avoid prematurely locking the seed to one bespoke house syntax,
- evaluate different seed geometries instead of assuming only scene JSON or full-grid tokens exist.

## One-shot vs two-pass

This correction should not force a false binary.

### Lane A — one-shot hybrid

The LLM emits one structured object containing:

- `semantic`
- `seedCode`
- optional `coarseVq`
- optional `providerLatents`
- `decoder`

This is the likely default lane because it is:

- cheaper,
- faster,
- simpler to operate,
- enough to test whether one-call hybrid visual coding is viable.

### Lane B — two-pass compile

Pass 1:

```text
User Prompt -> Semantic IR
```

Pass 2:

```text
Semantic IR -> Visual Seed Code / VQ hints
```

This is the **high-quality lane**:

- more expensive,
- more controllable,
- better for user review,
- better for recovery when one-shot seed quality is unstable.

### Current recommendation

- Keep **both** lanes legal.
- Treat **one-shot hybrid** as the default path to optimize first.
- Treat **two-pass compile** as the explicit high-quality lane.

## Adapter redefinition

The old adapter story in the image docs is too broad:

```text
scene semantics -> latent vocabulary
```

The corrected definition should be:

> The image adapter is primarily a **seed expander / visual-code compiler**. Its main job is to map compact visual seed representations into fuller decoder-native token grids. Semantic IR may still condition it, but semantic-only mapping is no longer the target architecture.

### Preferred priority order

The runtime should eventually converge on this order:

```text
1. providerLatents
2. coarse/full VQ hints
3. seedCode
4. semantic-only fallback
5. placeholder wiring path
```

This is important because it preserves the most research-interesting and most decoder-facing path first, while keeping the current conservative path as a fallback rather than the main story.

---

## What counts as a good seed representation

We should be careful here. The right move is probably **not** to invent a large custom visual language from scratch.

The strongest research-backed candidates are tokenizer-derived or hierarchy-derived seed layers:

- **TiTok-style compact tokens**
  - very small 1D visual token sequences
- **FlexTok-style prefix tokens**
  - variable-length ordered visual sequences
- **VAR-style coarse-scale tokens**
  - next-scale visual prediction
- **RQ-VAE-style residual-first codes**
  - first residual layer as seed, later layers as expansion target
- **SPAE-style lexical visual tokens**
  - more LLM-native symbolic visual codes

### Research implication

The repo should not lock "seed code = one bespoke syntax" too early.

Instead, the doctrine should lock:

- `Visual Seed Token` is first-class
- seed expansion is the adapter's main job
- the exact seed family remains a research variable until evaluation narrows it

## Manifest and CLI implications

If `Visual Seed Token` is first-class, then both manifest and CLI need explicit support.

### CLI

The user-facing surface should prefer:

- showing `Semantic IR`
- optionally summarizing the seed (`family`, `mode`, `length`, maybe preview)
- not dumping full unreadable VQ token grids by default

This creates a productive split:

- **human-facing layer:** semantic explanation
- **execution layer:** seed code / VQ hints / decoder tokens

### Manifest

The manifest should eventually record at least:

- `seedCode.family`
- `seedCode.mode`
- `seedCode.length`
- `seedCode.tokenizer`
- `seedCode.expansion.method`
- parse / validation outcome
- whether the run used:
  - `providerLatents`
  - `coarseVq`
  - `seedCode`
  - `semantic fallback`

This is part of the thesis. If image generation is about visible code compilation, the manifest has to preserve which code path actually ran.

## Not-to-do

This correction should also shrink the wrong directions.

### Do not

- treat `Semantic IR` as the permanent terminal image IR
- keep training only `scene-spec -> full latent` as the main line
- let the adapter read one giant prompt blob
- silently collapse back into "caption -> diffusion"
- invent a giant bespoke visual vocabulary before testing tokenizer-derived routes
- redefine the image thesis as a second image model

### Keep

- frozen decoder doctrine
- no diffusion / no second image path
- schema-first boundaries
- inspectable manifests
- small adapter / expander budget

## Current recommendation

Lock the following now:

1. `Semantic IR` remains in the canonical image route.
2. `Semantic IR` is no longer the primary image research object.
3. `Visual Seed Token` becomes first-class.
4. `one-shot hybrid` and `two-pass compile` are both valid lanes.
5. `two-pass compile` is the explicit high-quality lane.
6. The adapter is redefined primarily as a seed expander / visual-code compiler.

Leave the following open for further research:

- exact seed family
- default seed length
- exact coarse-grid size
- whether auxiliary semantic conditioning materially improves seed expansion

That is the smallest honest correction that restores the image route to the project's original hypothesis without over-locking the next implementation step.
